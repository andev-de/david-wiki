Q-106.379
Zentraler Nachrichtenversand mehrerer DvISE Installationen über einen DvISE PostMan


Frage
Problem Zentraler Nachrichtenversand mehrerer DvISE Installationen über einen DvISE PostMan
Produkt David
Priorität normal
Effekt normal
Datum 3.05.2007

Antwort
In diesem Konfigurationsbeispiel wird beschrieben wie DvISE für ein WAN über 3 Standorte konfiguriert wird, wobei nur ein Standort mit dem Internet verbunden ist. Die Anzahl der Standorte spielt dabei keine Rolle.Statt einer Verbindung zu einem Internet Service Provider (ISP) kann auch eine Anbindung an einen DvISE Host Provider erfolgen. In diesem Fall würde dann an Stelle der Wähl- oder Standleitung die im Server verwendete ISDN-Karte genutzt um via Internet Hosting Services (IHS) den eMail Versand, bzw. das Web-Publishing zu nutzen.

In unserem Beispiel gehen wir davon aus, das mehrere DvISE Installationen über einen Zentral installierten DvISE PostMan eMails versenden und empfangen. Für die nun folgende Konfiguration spielt es keine Rolle, ob das Server Betriebssystem Novell NetWare, Win32 oder Linux ist.

Die Hauptniederlassung Frankfurt hat eine Anbindung an einen ISP (Internet Service Provider). Die Zweigstellen München und Hamburg sind über ein WAN mit der Filiale in Frankfurt verbunden.

In Frankfurt wird eine "normale" DvISE Installation vorgenommen. Als Server wird hier ein Win32 Rechner incl. ServicePack, mit einem David verwendet. PostMan wird dann für den Betrieb an einer Wählleitung konfiguriert, d.h. hier werden über SMTP die eMail versendet und via POP3 vom Provider gepollt. die folgenden Einstellungen sollten in der Konfiguration vom DvISE PostMan über den DvISE Administrator vorgenommen werden.

Konfiguration PostMan

Menüpunkt: »POSTMAN -> SYSTEM -> GENERELL«

Tragen Sie für die Angabe SMTP HOST NAME (LOKAL) die IP Adresse des lokalen Servers ein. Unter DOMAIN NAME SERVER tragen Sie die als Bevorzugten DNS Server / Alternativen DNS Server die IP´s gemäß den Angaben Ihres ISP ein.

Menüpunkt: »POSTMAN -> SYSTEM -> PROVIDER«

Hier müssen Sie lediglich wie im Beispiel den Punkt GENERELL ÜBER PROVIDER (SMARTHOST) SENDEN aktivieren und als Mailserver des Providers die IP/DOMAIN Ihres ISP ein. Als Zeit für den Verbindungs AUFBAU wählen Sie je nach gewünschtem Polling Intervall Ihre Einstellung. In unserem Beispiel setzen wir diesen auf ALLE 10 MINUTEN. Als AKTIVER ZEITRAUM können Sie wie bereits beim Intervall die Einstellungen individuell wählen. In unserem Fall haben wir den AKTIVEN ZEITRAUM auf 00:00 - 00:00 gesetzt somit rund um die Uhr. Die Angabe für VERBINDEN AN haben wie hier auf JEDEN TAG gesetzt. Den Schalter MAILS SOFORT SENDEN und NACH PROGRAMMSTART VERBINDEN deaktivieren Sie.

Menüpunkt: »POSTMAN -> SYSTEM -> WEITERLEITEN«

Hier MÜSSEN Sie unbedingt den Punkt WEITERLEITEN aktivieren, damit die Nachrichten von anderen Zweigstellen entgegen genommen werden können. Wenn Sie den Punkt Ohne Einschränkungen aktivieren, so kann jeder User von außen über den DvISE PostMan eMails versenden also er ist konfiguriert als RELAY SERVER. Sie können hier aber im nach hinein noch Einschränkungen bei »erlaubte Weiterleitungen« setzen indem Sie dei Option »Ohne Einschränkungen« herausnehmen und in den Menüpunkt »Einschränkungen...« wechseln. Dort sollten Sie dann den Punkt »erlaubte Weiterleitungen« verwenden.

Menüpunkt: »POSTMAN -> DATENBANKEN -> ERLAUBTE WEITERLEITUNGEN«

Durch betätigen der EINF-Taste fügen Sie hier nun die IP Adressen der Remote PostMans aus den Niederlassungen München und Hamburg ein.Bei weiteren Standorten müssen für die »ERLAUBTEN WEITERLEITUNGEN« jeweils die IP Adressen der Server eingetragen werden, die über diese DvISE PostMan Installation Ihre eMails weiterleiten dürfen.

Konfiguration Verteilung

Nun müssen für die User der Niederlassungen Hamburg und München in der Filiale in Frankfurt Verteilungen eingerichtet werden, da diese User ja dort nicht als lokale Benutzer geführt werden und somit der User mit dieser Domain dort nicht bekannt ist. Daher geschieht die Verteilung nicht über die Benutzerkonfiguration, sondern über speziell dafür eingerichtete »VERTEILVORGABEN«.

Zuvor müssen Sie allerdings über den David Client ein Archive erstellen, in welches diese Maile verteilt werden sollen. In unserem Beispiel haben wir dieses Archive unterhalb von »ARCHIVE/GRUPPEN« mit dem Namen AUßENSTELLEN angelegt. Aktivieren Sie für dieses Archive über die Archive Eigenschaften den REMOTE ACCESS und vergeben Sie unter den PARAMETERN einen USERNAMEN und ein PASSWORT.

Im DvISE Administrator tragen Sie nun unter »DATENBANKEN -> VERTEILUNG -> VERTEILVORGABEN« bitte die eMail Adressen der User und Gruppen der Remote Server (Hamburg und München) mit dem zuvor angelegten Zielarchive ein.

Konfiguration des MailAccess Servers

Der Mail Access Server dient als reiner POP3 Server. Sie richten hierüber den Zugriff auf das o.g. Gruppen Archive ein. In der Systemkonfiguration sind für dieses Beispiel im MAServer keine weiteren Einstellungen notwendig.

In der ALIAS LISTE ist ein LOGIN ALIAS gleich dem o.g. Usernamen für das Archive AUßENSTELLEN anzugeben und das Gruppenarchive dieses Beispiels auszuwählen. Sie können hier auch definieren, dass die übertragenen Nachrichten (eMail, Fax, Voice etc.) nach der Übertragung gelöscht werden sollen und ob Dateianhänge gesendet werden sollen.

Konfiguration der Außenstellen - ohne Anbindung an einen ISP

An den Niederlassungen Hamburg und München wird die Konfiguration vom DvISE PostMan im DvISE Administrator wie folgt eingestellt:

Menüpunkt: »POSTMAN -> SYSTEM -> GENERELL«

Tragen Sie für die Angabe »SMTP HOST NAME (LOKAL)« die IP Adresse des lokalen Servers ein. Unter »DOMAIN NAME SERVER« tragen Sie die als Bevorzugten DNS Server / Alternativen DNS Server die IP´s gemäß den Angaben Ihres ISP ein. Wird kein Eintrag vorgenommen, so werden die eMails zwar versendet, aber der POP3 Request zum MAServer in Frankfurt wird nicht vorgenommen. Tragen Sie wenigstens ein fiktive IP ein.

Menüpunkt: »POSTMAN -> SYSTEM -> Provider«

Hier sollten Sie nun unter Mailserver des Providers die IP Adresse des Servers aus Frankfurt eintragen. Die Verbindungsintervalle können Sie beliebig setzen.

Menüpunkt: »POSTMAN -> SYSTEM -> WEITERLEITEN«

Bei diesem Punkt sollten Sie nun die Option »WEITERLEITEN« deaktivieren.

Menüpunkt: »POSTMAN -> SYSTEM -> ERWEITERT«

Hier sollte der Timeout auf 60 oder mehr gesetzt werden.

Menüpunkt: »POSTMAN -> DATENBANKEN -> DIREKTE WEITERLEITUNGEN«

Tragen Sie hier die eMail Adresse (mit Wildcard) der direkten Weiterleitungen sowie IP Adresse, bzw. Domain des Servers mit "PostMan am ISP" (Frankfurt) ein, damit eine direkte IP Verbindung hierzu aufgebaut wird und nicht eine Internet-Verbindung (ungewollt) zustande kommen kann.

Menüpunkt: »GRABBING SERVER -> DATENBANKEN -> POP3 POSTFÄCHER«

In diesem Menü tragen Sie neben der »BEZEICHNUNG« den Posteingang, also NAME/IP des Servers in Frankfurt ein.
Geben Sie als Benutzername/Passwort den USERNAMEN und das PASSWORT des o.g. Gruppenarchives (DAVID\ARCHIVE\GRUPPEN\AUßENSTELLEN) ein.

Die so empfangenen Nachrichten (eMail, Fax, Voice etc.) werden in den Zentraleingang gestellt. Folglich muss eine Verteilung eingerichtet werden. Dieses wird einmal in der Benutzerkonfiguration unter den Benutzerdaten über die E-Mail Adresse vorgenommen, die der Service Layer zur Verteilung auswertet. Für alle anderen Nachrichten wird ein Eintrag in der Verteildatenbank erforderlich.

Da die Struktur in diesem Beispiel sehr flach gehalten ist, können Sie auf dem Hauptserver (Servers mit "PostMan am ISP" in Frankfurt) mehrere Archive anlegen, die für User oder Gruppen gepollt werden. Sie können dann im MAServer die »TO-Adresse« beim Pollen mit der User E-Mail Adresse überschreiben lassen und somit direkt verteilen. Das macht Sinn beim Empfang und zu pollenden Faxen und Voice-Mails.

In der Konfiguration des DvISE Service Layer bei den Installationen mit "PostMan ohne ISP" (Hamburg und München) ist darauf zu achten, dass in der Systemkonfiguration keine Domain eingetragen ist. Der Sinn hierfür liegt darin, daß die User der einzelnen Installationen sonst keine Mails mehr untereinander versenden könnten, da der Service Layer die Domain auswerten würde und die Mail in den Zentraleingang stellen würde. Dieses soll hier nicht erreicht werden, sondern am Host (dem Server mit "PostMan am ISP" in Frankfurt). Dort tragen Sie die lokale Domain ein, die dann vom Service Layer in den Zentraleingang gestellt würde - hier greift dann aber wiederum die Verteilung und stellt die Mail in das Archive für den POP3 Account des Remote Server (die Server mit "PostMan ohne ISP"). Somit ist auch ein "interner" Mailversand zwischen den Standorten möglich.
